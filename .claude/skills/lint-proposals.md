# Lint Proposals Skill

Audit `docs/11-proposals/` for status drift, missing frontmatter, misplaced
files, and convention violations. The skill defines the rules; the
[`/mxcli-dev:proposal-lint`](../commands/mxcli-dev/proposal-lint.md)
command is the trigger.

The skill produces a **report**. Cleanup actions (moves, frontmatter
migration) happen only with explicit user approval, in a second turn.

## Frontmatter convention

Every proposal must have YAML frontmatter at the top of the file:

```yaml
---
title: <short title>
status: <status>          # see vocabulary below
date: YYYY-MM-DD
author: <optional>
related:                  # optional
  - <PR # or proposal slug>
supersedes: <proposal>    # optional, if this replaces another
superseded-by: <proposal> # optional, if a newer proposal exists
shipped-in: <version>     # optional, when status: done
---
```

This matches the wiki and ADR conventions — one frontmatter style across
the repo.

## Status vocabulary (canonical)

Exactly one of these lowercase values:

| Status | Meaning | Folder |
|--------|---------|--------|
| `draft` | Early design, not finalised | active |
| `proposed` | Design complete, not yet started | active |
| `partial` | Some phases implemented, more to come | active |
| `done` | Fully implemented | `archive/` |
| `superseded` | Replaced by another proposal | `archive/` |
| `abandoned` | Explicitly not pursuing | `archive/` |
| `reference` | Not a proposal — durable reference doc | move **out** of `11-proposals/` |

**`implemented` is not valid** — normalise to `done`. The legacy README
status `Done` maps directly to frontmatter `done`.

## Lint rules

For each file under `docs/11-proposals/` (recursive):

| Rule | Check | Action on violation |
|------|-------|---------------------|
| R1 | YAML frontmatter present with `title`, `status`, `date` | Flag; propose migration from inline `**Status:**` if present |
| R2 | `status` is in canonical vocabulary | Flag; normalise `Implemented` → `done` |
| R3 | Status matches location (archived statuses in `archive/`, reference moved out, active in root) | Flag; propose `git mv` |
| R4 | Every file appears in the README index; every README entry resolves to a real file | Flag both directions |
| R5 | No duplicate content across folders (notably the root `/proposals/` directory) | Flag |
| R6 | Supersession is symmetric: A `superseded-by: B` ⇒ B `supersedes: A` and at least one in `archive/` | Flag |
| R7 | Filename follows `proposal-<kebab-case>.md` (informational only — not enforced retroactively) | Flag for future normalisation |

## Procedure

When invoked:

1. **Walk** `docs/11-proposals/` and `docs/11-proposals/archive/`.
2. **Check R1–R7** against every file. Also scan `/proposals/` at the
   repo root for stragglers (R5).
3. **Group findings** into report categories:
   - `frontmatter-missing` — R1 failures
   - `status-noncanonical` — R2 failures (e.g. `Implemented`)
   - `move-to-archive` — active proposals with terminal status
   - `move-out` — `reference`-status files in proposals folder
   - `readme-drift` — R4 failures
   - `duplicate` — R5 failures
   - `supersession-asymmetric` — R6 failures
   - `naming` — R7 (informational, low priority)
4. **Output the report.** Group by category; for each finding, propose
   the concrete action (`git mv X archive/X`, `add frontmatter`, etc.).
5. **Stop.** Do not apply changes. Wait for user direction.

## First-run migration

The current state uses inline `**Status:** Draft` instead of YAML
frontmatter. The first lint run will produce a large
`frontmatter-missing` category. Migration rules:

1. **Derive `status:`** from the inline `**Status:**` line if present.
2. **Derive `date:`** from the inline `**Date:**` line if present.
3. **Derive `author:`** from inline `**Author:**` if present.
4. **Derive `title:`** from the first `# ` heading.
5. **Preserve the inline lines** for one cycle (belt-and-braces) — drop
   them in a follow-up pass once frontmatter is confirmed stable.
6. **Flag for manual triage** any file where inline status is missing
   *and* the README has no entry for it.

After migration, the README index becomes mechanically derivable from
frontmatter; a follow-up slice can switch it to a generated artifact.

## What this skill is NOT for

- **Approving or rejecting proposals.** That's the maintainer's call,
  recorded by changing `status:` — the lint only checks consistency.
- **Renaming files retroactively.** R7 is informational; existing files
  keep their names.
- **Editing proposal content.** Body changes are out of scope; lint
  touches frontmatter and location only.
- **Generating the README.** The README index is regenerated by
  `scripts/gen_proposals_readme.py` (everything below the `AUTOGEN`
  marker). Run it after any status change; the hand-written conventions
  and status-legend header above the marker are preserved. The lint only
  flags drift — it does not hand-edit the index.
