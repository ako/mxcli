# Proposal: Deterministic AI-Capability Dataset & Report Generator

**Status:** Draft
**Date:** 2026-06-22
**Author:** AI-assisted design

---

## Problem Statement

We publish an *AI Capabilities* report (`ai-capabilities-report-<version>.html`) that
compares what each Studio Pro AI tooling setup can do to a Mendix app across six columns:

| Column | What it is |
|--------|-----------|
| **Maia** | Studio Pro's built-in agent (native, in-IDE) |
| **3rd-party + MCP** | external agent via the PED MCP server (SP 11.11+) |
| **3rd-party + MCP + Concord** | the above plus the Concord extension's tools |
| **mxcli (file)** | mxcli on a closed `.mpr` |
| **mxcli + MCP** | mxcli routing through the live PED server |
| **mxcli + MCP + Concord** | live PED + Concord gap-fillers |

Today that report is **assembled by an LLM** reading `MDL_FEATURE_MATRIX.md` and
`PED_MCP_CAPABILITIES.md`. The result is plausible but **not accurate and not
reproducible**: cells drift from reality, parent/child rows contradict each other,
and every new Studio Pro release means another manual re-write with no audit trail.

We want the inverse: a **deterministic dataset** — produced by *running probes*, not by
prose — that records, per capability and per tooling setup, (a) the status and (b) the
**evidence** that produced it. The report is then *rendered from* that dataset. The whole
dataset is regenerated by a single reproducible run against a version-matched environment,
so each Studio Pro release yields a new, diffable dataset.

**Non-goals.** This is not a quality/regression eval (how *good* generated apps are — see
[`proposal-eval-framework.md`](proposal-eval-framework.md)) and not a microflow test DSL
(see [`proposal-mdl-test-framework.md`](proposal-mdl-test-framework.md)). It answers a
narrower, factual question: *"Is capability X possible via setup Y, and what proves it?"*

## Key Insight: derive 6 columns from 3 probe channels

The six report columns are not six independent things to measure. They collapse to **three
probe channels** plus one manual overlay:

| Channel | What it probes | Mechanism |
|---------|----------------|-----------|
| `file` | mxcli on the closed `.mpr` | run an MDL probe → `mxcli check` → execute → `mx check` |
| `ped`  | the Studio Pro MCP (PED) server | `tools/list` + create-whitelist + attempt op → `ped_check_errors` |
| `concord` | the Concord MCP server | `tools/list` + attempt op |

The columns are then **derived by rule** from the channels:

```
maia          = overlay (manual, flagged unverified)   ← see "Maia column" below
3rd-party+MCP = ped
3rd-party+MCP+Concord = ped ∪ concord
mxcli (file)  = file
mxcli+MCP     = ped         (+ mxcli client gap-fillers: file-mode reads while live)
mxcli+MCP+Concord = ped ∪ concord  (+ mxcli gap-fillers)
```

This is the heart of the proposal. Because the derivation is a **pure function** of channel
results, the matrix is internally consistent *by construction* — the contradictions in the
hand-assembled report (a child `y` under a parent `n`, two MCP columns disagreeing for no
reason) simply cannot occur.

### The Maia column

Maia is an in-IDE LLM agent with no external interface to enumerate, so it cannot be probed
deterministically. Per the design decision for this proposal, Maia stays a **hand-curated
overlay**: a per-version `maia-overlay-<version>.yaml` whose every entry is tagged
`source: overlay` and surfaced in the report as *manually asserted, not machine-verified*.
The dataset linter (below) refuses overlay entries that lack a one-line justification, so the
honesty of that column is enforced, not assumed.

## Architecture

```
capabilities.yaml            ← the capability registry (single source of truth: the rows)
maia-overlay-<ver>.yaml      ← hand-curated Maia column, flagged unverified
        │
        ▼
mxcli capabilities probe  --project <fixture-ver>.mpr --ped-url … --concord  --maia-overlay …
        │  (one reproducible run against a live, version-matched environment)
        ├── file channel:    reset fixture → mxcli check → execute MDL → mx check
        ├── ped channel:     tools/list + ped_get_schema whitelist + attempt op + ped_check_errors
        ├── concord channel: tools/list + attempt op
        ├── derive 6 columns from channels (pure function)
        └── merge Maia overlay
        ▼
datasets/capabilities-<ver>.json   ← committed, diffable, every cell carries evidence
        │
        ├── mxcli capabilities lint   → internal-consistency + provenance checks
        └── mxcli capabilities render → ai-capabilities-report-<ver>.html
```

### 1. The capability registry (`capabilities.yaml`)

The rows of the report become a declarative registry — the *spec* of what we measure. Each
capability is one entry:

```yaml
- id: domain-model.persistent-entity.create     # stable, kebab; survives relabeling
  section: Domain Model
  label: Create entity
  level: 2
  parent: domain-model.persistent-entities
  probe:
    file:    { script: probes/domain/create-entity.mdl }     # MDL that exercises it
    ped:     { op: ped_create_document, type: "DomainModels$Entity" }
    concord: { tool: delete_document }                        # presence ⇒ available
```

A capability with no `probe.file` (e.g. *Conversational in-IDE chat*) is `n/a` for that
channel — recorded explicitly, never silently. This registry replaces the prose rows of
`MDL_FEATURE_MATRIX.md` as the authority for *which* capabilities exist; the matrix doc
becomes a rendered view.

### 2. Probe channels

**`file`** — reuses the existing doctype-test / `mxcli test` machinery. For each capability:
reset the fixture to a known baseline (git checkout of the v2 `mprcontents`, or a copy),
`mxcli check` the probe MDL (syntax + anti-patterns), execute it, then `mx check` the project.
Status: `y` = executes and validates clean · `w` = executes but `mx check` errors (broken) ·
`n` = grammar/executor rejects or not implemented · `r` = SHOW/DESCRIBE only.

**`ped`** — promote `cmd/mcpprobe` into a reusable client library (`mdl/backend/mcp` already
speaks PED; the probe needs the raw handshake `mcpprobe` already implements). Two layers:
1. *Static surface* — dump `tools/list` and the create whitelist via `ped_get_schema`. Many
   cells are gated purely by whether a `$Type` is on the whitelist; that is directly
   enumerable (e.g. `JavaActions$JavaAction`, `Microflows$Nanoflow` are off it → `n`).
2. *Operational* — attempt the real op (`ped_create_document` / `ped_update_document` /
   `pg_write_page`), then `ped_check_errors`. Accept + clean = `y` · rejected = `n` (capture
   the exact server error string as evidence) · accepted but errors = `w`.

**`concord`** — dump Concord's `tools/list`; tool presence ⇒ capability (read-only tools →
`r`). Operational probe where it is safe to mutate the fixture.

Every probe records **evidence**: the command/op invoked, the raw response or error, and the
environment fingerprint. No cell exists without a trace back to what produced it.

### 3. Dataset (`capabilities-<ver>.json`)

```json
{
  "schemaVersion": 1,
  "generatedAt": "2026-06-22T…",
  "environment": {
    "studioPro": "11.12.0", "ped": "1.0.0", "concord": "0.x",
    "mxcli": "<git commit>", "fixture": "capability-fixture-11.12.mpr"
  },
  "capabilities": [
    {
      "id": "domain-model.persistent-entity.create",
      "section": "Domain Model", "label": "Create entity", "level": 2,
      "parent": "domain-model.persistent-entities",
      "channels": {
        "file":    { "status": "y", "evidence": { "probe": "probes/domain/create-entity.mdl", "mxCheck": "0 errors" } },
        "ped":     { "status": "y", "evidence": { "op": "ped_create_document", "checkErrors": 0 } },
        "concord": { "status": "n/a" }
      },
      "columns": {
        "maia":         { "status": "y", "source": "overlay", "note": "native domain-model authoring" },
        "tpMcp":        { "status": "y", "source": "ped" },
        "tpMcpConcord": { "status": "y", "source": "ped∪concord" },
        "file":         { "status": "y", "source": "file" },
        "mxMcp":        { "status": "y", "source": "ped" },
        "mxMcpConcord": { "status": "y", "source": "ped∪concord" }
      }
    }
  ]
}
```

The committed JSON is the artifact that makes per-version reproduction real: `git diff
capabilities-11.11.json capabilities-11.12.json` *is* the changelog of what a Studio Pro
release changed for AI tooling.

### 4. Dataset linter (`mxcli capabilities lint`)

A meta-check that runs on the emitted JSON and fails CI on:
- **Monotonicity** — a child capability more capable than its parent (the contradiction class
  in today's report).
- **Rollup** — a parent's status is the correct aggregate of its children.
- **Coverage** — every registry entry has either a probe result or an explicit `n/a`.
- **Provenance** — every `source: overlay` (Maia) cell carries a justification note.
- **Freshness** — `environment` matches the requested version; no stale evidence.

### 5. Renderer (`mxcli capabilities render`)

Reuses the existing self-contained HTML template (the tree/tooltip UI is good). It injects the
`R[]` rows and tooltips straight from the dataset, so the published report is a *pure function
of the dataset* — no LLM in the rendering path. Markdown and JSON output modes feed
`MDL_FEATURE_MATRIX.md` and any docs-site page from the same source.

## Reproducing per Studio Pro version

A pinned environment manifest per version (fixture project at that version — committed, or
`mxcli new --version`; PED + Concord versions; mxcli commit). One command regenerates the
entire dataset against the live, version-matched environment:

```bash
mxcli capabilities probe \
  --registry      docs/01-project/capabilities.yaml \
  --project       reference/capability-fixture-11.12.mpr \
  --ped-url       http://localhost:7782/mcp \
  --concord \
  --maia-overlay  docs/01-project/maia-overlay-11.12.yaml \
  --out           datasets/capabilities-11.12.json

mxcli capabilities lint   datasets/capabilities-11.12.json
mxcli capabilities render datasets/capabilities-11.12.json --format html \
  > ai-capabilities-report-11.12.html
```

Onboarding a new minor = bump the fixture, point at the new live Studio Pro + PED, refresh the
Maia overlay, re-run. The diff against the previous dataset is the per-version delta.

## Implementation Plan

### Files to create

| File | Change |
|------|--------|
| `docs/01-project/capabilities.yaml` | The capability registry (rows + probe specs) |
| `docs/01-project/maia-overlay-<ver>.yaml` | Hand-curated Maia column, per version |
| `probes/**/*.mdl` | File-channel probe scripts (many reuse `mdl-examples/doctype-tests/`) |
| `cmd/mxcli/cmd_capabilities.go` | `mxcli capabilities probe \| lint \| render` |
| `internal/capabilities/registry.go` | Load/validate the registry |
| `internal/capabilities/probe_file.go` | File channel (wraps check + execute + `mx check`) |
| `internal/capabilities/probe_ped.go` | PED channel (promote `cmd/mcpprobe` to a library) |
| `internal/capabilities/probe_concord.go` | Concord channel |
| `internal/capabilities/derive.go` | Pure channel→column derivation + rollup |
| `internal/capabilities/lint.go` | Dataset consistency/provenance linter |
| `internal/capabilities/render_html.go` | Dataset → HTML (existing template), + md/json |
| `datasets/capabilities-<ver>.json` | Committed output, one per Studio Pro version |

### Phasing

- **Phase 1 — file channel + registry + dataset + linter.** Deterministically nails the
  entire **mxcli (file)** column (the most authoritative one) with zero live dependencies.
  Reuses the doctype-test harness. Ships the registry schema and the JSON contract.
- **Phase 2 — `ped` channel + column derivation.** Promote `cmd/mcpprobe` to a library;
  enumerate the whitelist; attempt ops + `ped_check_errors`. Lights up the four MCP columns.
- **Phase 3 — `concord` channel + Maia overlay + HTML renderer.** First fully-rendered report
  generated from the dataset, replacing the AI-assembled HTML.
- **Phase 4 — per-version regen workflow.** Pinned fixtures, committed datasets, a `make
  capabilities-<ver>` target, and a cross-version diff view.

## Open Questions

1. **Fixture strategy** — one shared fixture project per version, or per-section fixtures to
   keep probes isolated? (Leaning: one fixture, reset between probes.)
2. **Concord operational probes** — Concord's `save_all`/`run_app` are known-broken in 11.11
   Beta; do we probe them operationally (capturing the breakage as `w` evidence) or stop at
   tool-presence? (Leaning: probe operationally — breakage *is* the finding.)
3. **mxcli client gap-fillers** — exact rule for how `mxcli+MCP` differs from `3rd-party+MCP`
   (file-mode reads while live, skills). Encode as an explicit derivation rule with evidence.
4. **Where datasets live** — `datasets/` at repo root vs `docs/14-capabilities/`.
